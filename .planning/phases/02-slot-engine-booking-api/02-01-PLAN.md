Export these constants so tests can reference them if needed.

Algorithm for `computeAvailableSlots(input: SlotEngineInput): string[]`:

1. Parse `input.date` as a local date in Europe/Lisbon timezone.
2. Compute `businessStart` = date at 09:00 Lisbon time, `businessEnd` = date at 18:00 Lisbon time.
3. Generate candidate start times on 30-min grid from businessStart to businessEnd (exclusive):
   - Candidates: 09:00, 09:30, 10:00, ..., 17:30 (all in Lisbon local time)
4. For each candidate, compute `candidateEnd = candidate + durationMin` and `requiredEnd = candidate + durationMin + BUFFER_MIN`.
   - If `requiredEnd > businessEnd` (i.e., the booking plus buffer would exceed 18:00), discard this candidate.
5. Filter `existingBookings` to only those with `status === 'confirmed'`.
6. For each confirmed booking, compute its occupied window: `[bookingStart, bookingEnd + BUFFER_MIN)`.
   - Convert `start_time` and `end_time` from ISO strings to Date objects.
7. For each remaining candidate, check overlap with ALL occupied windows:
   - Overlap exists if `candidateStart < occupiedEnd AND candidateEnd > occupiedStart`
   - Where `candidateEnd = candidateStart + durationMin` (the service end, NOT including buffer — buffer is only for the gap after the service)
   - If overlap with ANY occupied window, discard the candidate.
8. Convert surviving candidates to ISO 8601 strings and return sorted.

For timezone handling: Use `Intl.DateTimeFormat` with `timeZone: 'Europe/Lisbon'` to resolve the correct UTC offset for the given date, OR use date-fns-tz if needed (install it if date-fns alone is insufficient). The key requirement: `new Date('2026-01-15T09:00:00')` must resolve to 09:00 Lisbon time (which is 09:00 UTC in winter, 08:00 UTC in summer).

One approach that avoids external timezone deps: construct dates using explicit offset. For a given date, determine if it falls in DST (last Sunday of March to last Sunday of October in Portugal). Winter offset = +00:00, Summer offset = +01:00. Build ISO strings like `2026-01-15T09:00:00+00:00` or `2026-06-15T09:00:00+01:00`.
